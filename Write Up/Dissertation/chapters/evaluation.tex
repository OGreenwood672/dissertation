\section{Evaluation Setup}
\label{sec:4.1}

\textbf{Environment Setup}
\label{sec:4.1.1}

For the following evaluations, a specific cooperative simulation setup is created.
There exist two agents.
Each agent has two ingredients to collect from unique stations.
They also each have a station to drop off their product.
Each agent cannot observe their own pick-up stations, but can observe their teammates at an infinite range.
This setup ensures cooperation is required and, therefore, communication must emerge for optimal task reward.

The world is defined as a 5x5 grid.
This is chosen to be small enough to minimise the spatial navigation problem, allowing the evaluation to focus on the emergent language.
For the 5x5 grid, agents with no communication must take on expectation $\sim$ 35 actions to find both stations (see Appendix \ref{sec:num-actions}).
Therefore, to reduce the difference in mean task reward per step between the communication and no-communication protocols, the number of actions agents are allowed to take in each world is limited to 30.

\textbf{MAPPO Hyperparameters}
\label{sec:4.1.2}

The use of standardised hyperparameters is preferred over aggressive grid searches to demonstrate the robustness of the AI Mother Tongue architecture.

A batch size of 512, 57,600 datapoints per update phase (see section \ref{sec:4.2.1}) and four optimisation epochs are selected to optimise stability and sample efficiency.
The batch size of 512, a common power-of-two choice in machine learning, reduces gradient variance for LSTM-based actor-critic networks whilst maximising GPU utilisation.
The low four epochs prevent overfitting to stale data during update phases, aligning with PPO recommendation of 3-10 epochs \cite{ppo_bestpractices}.

\textbf{Communication Hyperparameters}
\label{sec:4.1.3}

These communication hyperparameters apply to all communication protocols.
A vocabulary size of 32 is chosen to be deliberately small compared to the theoretical number of possible external observations to encourage information compression.
Each code in the codebook is of size eight to allow sufficient spatial encoding whilst also respecting the reality of limited bandwidth.

\section{System Performance}
\label{sec:4.2}

Reinforcement Learning with multiple agents requires millions of datapoints for policy updates and an equal number for evaluating the language and policy.
This section quantitatively measures the performance gains through the custom systems developed.
Specifically, it analyses the steps-per-second of the Rust simulation developed and the memory efficiency of the \lstinline[language=Python]|RolloutBuffer|.
Benchmarks were evaluated on NVIDIA RTX 5060 Ti and AMD Ryzen 7 3700X.

\subsection{Vectorised Environment Throughput}
\label{sec:4.2.1}

A major issue with Python's concurrent threading is the GIL.
While Python's built-in library \lstinline[language=Python]|multiprocessing| can, in theory, bypass this limitation, it introduces considerable overhead.
For this RL project, all threads require synchronisation at every simulation timestep.

To justify developing the simulation in Rust rather than Python, whilst evaluating C1, a simple mock simulation was developed in Python.
This mock simulation was compared to the vectorised Rust backend.
Environment steps per second (SPS) were calculated to analyse the throughput of each simulation.
Number of worlds parallised was scaled ($W \in {2^0, 2^1, …, 2^{12}}$) to measure concurrency efficiency.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{sim-comparison.png}
    \caption{Concurrent Environment Throughput Performance. Comparison of Python multiprocessing and Rust with Rayon across an exponentially increasing number of parallel worlds}
    \label{fig:sim-time-comparison}
\end{figure}

As shown in figure \ref{fig:sim-time-comparison}, the Python baseline has 100x less SPS than the Rust simulation.
Despite being able to use multiple CPU cores, the serialisation and the Inter-Process Communication overhead required to transfer the necessary data between isolated processes at every time step heavily bottleneck the throughput.

In contrast, Rust is able to utilise \lstinline[language=Rust]|Rayon|'s thread-pool to efficiently manage the parallel world steps.
Throughput scales with the number of worlds parallelised, reaching $239,007$ SPS.
By using Rust's native Rayon threading, the sequential fraction of the program is minimised per Amdahl's Law \cite{amdahl}, allowing near-perfect scaling that saturates the CPU cores.

For the evaluation, $960$ worlds were parallelised across a single rollout phase.
This maximised the parallisation whilst also collecting an apt amount of datapoints for the update phase ($960 * 30 * 2 = 57,600$).

\subsection{Rollout Buffer Efficiency}
\label{sec:4.2.2}

A secondary major potential bottleneck is transferring the results of the rollout phase to the training phase (across the CPU-GPU boundary).
To evaluate the \lstinline[language=Python]|RolloutBuffer|, the memory allocation and transfer overhead are compared to a baseline method (appending to Python lists and executing \lstinline[language=Python]|torch.stack| after the rollout phase).

Whilst there is a $\sim 5\%$ improvement in allocation time when using Python's dynamic list, the large improvement in VRAM will enable scaling this method to millions of datapoints.

Reconstruction loss should fall between 0.08 and 0.15.
Commitment Loss must remain below 1.0.

By the end of the training, the VQ-VAE, the commitment loss has converged to $\sim 0.25$.

\section{Cognitive Mapping and Spatial Memory}
\label{sec:4.6}

\subsection{Memory Probing}
\label{sec:4.6.1}

To understand what the agent concretely knows, six strict linear probes were developed following Guillaume et al.'s methodology \cite{Probing}.
Memory is tested for allocentric (information stored using a global coordinate system) and egocentric (information stored relative to the self) memory.
Each probe takes as input the final layer of the Long-Short Term Memory (LSTM) module, and predicts one of the following using MSE loss:
\begin{itemize}
    \item The agent's own next target's coordinates (tests for allocentric memory)
    \item The agent's partner's next target's coordinates(tests for allocentric memory)
    \item The agent's own next target's relative coordinates from the agent's coordinates (tests for egocentric positional memory)
    \item The agent's partner's next target's relative coordinates from the agent's coordinates (tests for egocentric positional memory)
    \item A unit vector in the direction of the agent's next target (tests for egocentric directional memory)
    \item A unit vector in the direction of the agent's partner's next target (tests for egocentric directional memory)
\end{itemize}

Initially, in a simulation run, agents have no way of knowing their targets' locations.
To account for this, the models are trained only on the last half of the timesteps to ensure the LSTM contains the necessary information for training.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{lstm_probing.png}
    \caption{LSTM Probing Models Test Loss over simulation runs with various tasks}
    \label{fig:lstm-probing}
\end{figure}

The relative distance and unit direction models are shown to never improve over the course of a simulation, suggesting the LSTM does not follow egocentric memory.
However, the absolute location model shows that the LSTM uses allocentric memory.

Even though the absolute location model shows the LSTM storing some information, it converges to an MSE of 1.6.
This error represents an uncertainty radius of 1.6 cells, showing how the memory acts as a 'regional beacon' for the agent's targets.

An important conclusion from figure \ref{fig:lstm-probing} is that the agent's need for communication decreases as the simulation runs, and they store target locations in the LSTM module.
