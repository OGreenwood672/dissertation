\section{Evaluation Setup}
\label{sec:4.1}

\textbf{Environment Setup}
\label{sec:4.1.1}

For the following evaluations, a specific cooperative simulation setup is created.
There exist two agents.
Each agent has two ingredients to collect from unique stations.
They also each have a station to drop off their product.
Each agent cannot observe their own pick-up stations, but can observe their teammates with an infinite range of view.
This setup ensures cooperation is required and, therefore, communication must emerge for optimal task reward.

The world is defined as a 5x5 grid.
This is chosen to be small enough to minimise the spatial navigation problem, allowing the evaluation to focus on the emergent language.
For the 5x5 grid, agents with no communication must take on expectation $\sim$ 35 actions to find both stations (see Appendix \ref{sec:num-actions}).
Therefore, to elevate the difference in mean task reward per step for communication and no communication protocols, the number of actions agents are allowed to take in each world is limited to 30.

\textbf{MAPPO Hyperparameters}
\label{sec:4.1.2}

The use of standardised hyperparameters is chosen rather than performing aggressive grid searches to demonstrate the robustness of the AI Mother Tongue architecture.

A batch size of 512, 57,600 datapoints per update phase (see section \ref{sec:4.2.1}) and four optimisation epochs are selected to optimise stability and sample efficiency.
The batch size of 512, a common power of two choice in machine learning, reduces gradient variance for LSTM-based actor-critic networks whilst maximising GPU usage.
The low four epochs prevent overfitting to stale data during update phases, aligning with PPO recommendation of 3-10 epochs \cite{ppo_bestpractices}.

\textbf{Communication Hyperparameters}
\label{sec:4.1.3}

These communication hyperparameters apply to all communication protocols.
A vocabulary size of 32 is chosen to be deliberately small compared to the theoretical number of possible external observations to encourage information compression.
Each code in the codebook is of size eight to allow sufficient spatial encoding whilst also respecting the reality of limited bandwidth.

\section{System Performance}
\label{sec:4.2}

Reinforcement Learning with multiple agents requires millions of datapoints for updating the policies and an equal number to evaluate the language and policy.
This section quantitatively measures the performance gains through the custom systems developed.
Specifically, it analyses the steps per second of the Rust simulation developed and the memory efficiency of the \lstinline[language=Python]|RolloutBuffer|.
Benchmarks were evaluated on NVIDIA RTX 5060 Ti and AMD Ryzen 7 3700X.

\subsection{Vectorised Environment Throughput}
\label{sec:4.2.1}

A major issue with Python running multiple threads concurrently is the GIL.
While Python's built-in library \lstinline[language=Python]|multiprocessing| can, in theory, bypass this limitation, it introduces considerable overhead.
For this RL project, all threads require synchronisation at every simulation timestep.

To justify developing the simulation in Rust rather than Python, whilst evaluating C1, a simple mock simulation was developed in Python.
This mock simulation was compared to the vectorised Rust backend.
Environment steps per second (SPS) was calculated to analyse the throughput of each of the simulations.
Number of worlds parallised was scaled ($W \in \{2^0, 2^1, ..., 2^{12}\}$) to measure concurrency efficiency.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{sim-comparison.png}
    \caption{Concurrent Environment Throughput Performance. Comparison of Python multiprocessing and Rust with Rayon across an exponentially increasing number of parallel worlds}
    \label{fig:sim-time-comparison}
\end{figure}

As shown in figure \ref{fig:sim-time-comparison}, the Python baseline has 100x less SPS than the Rust simulation.
Despite being able to use multiple CPU cores, the serialisation and the Inter-Process Communication overhead required to transfer the necessary data between isolated processes at every time step heavily bottleneck the throughput.

In contrast, Rust is able to utilise \lstinline[language=Rust]|Rayon|'s thread-pool to efficiently manage the parallel world steps.
Throughput scales with the number of worlds parallelised, reaching $239,007$ SPS.
By using Rust's native Rayon threading, the sequential fraction of the program is minimised per Amdahl's Law, allowing near-perfect scaling that saturates the CPU cores.

For the evaluation, $960$ worlds parallelised over a single rollout phase were chosen.
This maximised the parallisation whilst also collecting an apt amount of datapoints for the update phase ($960 * 30 * 2 = 57,600$).

\subsection{Rollout Buffer Efficiency}
\label{sec:4.2.2}

A secondary major potential bottleneck is transferring the results of the rollout phase to the training phase (across the CPU-GPU boundary).
To evaluate the \lstinline[language=Python]|RolloutBuffer|, the memory allocation and transfer overhead are compared to a baseline method (appending to Python lists and executing \lstinline[language=Python]|torch.stack| after the rollout phase).

\begin{table}[h]
    \centering
    \begin{tabular}{|l|c|c|c|}
    \hline
    \textbf{Memory Strategy} & \textbf{Allocation Time (s)} & \textbf{Peak VRAM Overhead (MB)} \\ \hline
    \textbf{Dynamic Python Lists} & $6.47 \pm 0.18$ & $1213.45$ \\ \hline
    \textbf{Pre-allocated RolloutBuffer} & $6.79 \pm 0.23$ & $598.23$ \\ \hline
    \end{tabular}
    \caption{Memory Management Overhead per Epoch: $T=20,000$, $W=32$}
    \label{tab:rolloutbuffer-eval}
\end{table}

As shown in \ref{tab:rolloutbuffer-eval}, the \lstinline[language=Python]|RolloutBuffer| method reduces VRAM by over $50\%$.
This is because Python's dynamic arrays often require copying their contents from one memory location to another to increase their size.
Whilst there is a $\sim 5\%$ improvement in allocation time when using Python's dynamic list, the large improvement in VRAM will enable scaling this method to millions of datapoints.


\section{Offline Pre-training}
\label{sec:4.3}

Before we are able to integrate the language into reinforcement learning, its health must be evaluated.
A poorly initialised frozen encoder could limit the agents' language considerably, preventing them from reaching their maximum cooperation potential.
These results contribute to C4.

\subsection{Discretisation Stability: VQ-VAE vs. SQ-VAE}
\label{sec:4.3.1}

For this evaluation, reconstruction loss and commitment loss are tracked throughout a training process of 50 epochs using 26880 datapoints.
Liu et al. defined a healthy encoder for the AI Mother Tongue system.
Reconstruction loss should fall between 0.08 and 0.15.
Commitment Loss must remain below 1.0.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{recon_loss_eval.png}
    \caption{Reconstruction Loss over 50 epochs comparing VQ-VAE and SQ-VAE}
    \label{fig:vae_training_eval}
\end{figure}

As shown in \ref{fig:vae_training_eval}, both methods of training got below the maximum reconstruction loss of 0.15.
However, training using SQ-VAE lead to a final reconstruction of below 0.08.
Liu et al.'s recommended loss for a healthy codebook is based on low reconstruction, often being a symptom of 'mode collapse'.
This is where the network memorises a small set of generalisable states.

To determine if this 'mode collapse' has occurred, the codebook utilisation was analysed.

\begin{table}[h]
    \centering
    \begin{tabular}{|l|c|c|}
    \hline
    \textbf{} & \textbf{VQ-VAE} & \textbf{SQ-VAE} \\ \hline
    \textbf{Codebook Utilisation} & $66.67\% \pm 4.8\%$ & $94.8\% \pm 6.5\%$ \\ \hline
    \end{tabular}
    \caption{Codebook utilisation comparison between training with VQ-VAE and SQ-VAE}
    \label{tab:codebook_util}
\end{table}

The table \ref{tab:codebook_util} clearly shows that using an SQ-VAE, though having a low reconstruction loss, in fact has a better codebook utilisation than using a VQ-VAE.
Therefore, this low loss indicates high effectiveness in the model rather than 'mode collapse'.

By the end of the training, the VQ-VAE, the commitment loss has converged to $\sim 0.25$.
This is well below Liu et al.'s requirement.

\subsection{Baseline Comparisons}

To ascertain the effectiveness of the AI Mother Tongue architecture, it is compared against two other protocols.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{reward_mean_comparison.png}
    \caption{Reward mean per step over 30,000 PPO training steps. Each protocol was trained across three runs, and a 95\% confidence interval is shown.}
    \label{fig:reward-mean-comparison}
\end{figure}

As shown in Figure \ref{fig:reward-mean-comparison}, the AI Mother Tongue architecture was able to outperform both other protocols.
The No Communication protocol showed the requirement for communication in the simulation; this protocol converged on a reward mean per step of 0.09.
The Discrete Comms protocol showed a struggle to communicate any useful information from one agent to another and had a similar training convergence to the No Communication protocol, suggesting it experienced the "Joint Exploration Problem".

The AI Mother Tongue protocol demonstrates two distinct phases in its training.
The first phase underperforms all the other protocols, as it struggles to balance optimising its auxiliary losses and the standard training losses.
The second phase converges to 0.13 reward mean per step as it learns to use the information stored in order to optimise the auxiliary heads to optimise the standard training losses.

No protocol is able to reach the 0.24 reward mean per step recorded by the expert agent.
This is due to a limitation the agents have.
The agents can only view the nearest entity (that isn't their own pick-up station), whereas the expert agent has a pure, global view of the environment.
This means other entities often hide other, more important entities, preventing agents from matching this expert agent's score.
Therefore, the 0.13 reward demonstrates that the agents successfully utilised the AI Mother Tongue protocol to overcome the POMDP.

Training MARL policies often requires 10x or 100x the number of steps than 30,000 steps.
However, for this evaluation, 30,000 steps is sufficient as the core research focus - the spontaneous semantic compression emerged within the first 12,000 steps (see section \ref{sec:4.5.1}).

\section{Information-Theoretic \& Linguistic Properties}

\subsection{Spontaneous Semantic Compression}
\label{sec:4.5.1}

Throughout training, the communication perplexity of the agent's vocabulary is tracked.
Communication perplexity represents how uncertain the agent is about the next token, measured as $e^{-\text{entropy of vocabulary}}$.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{comm_perplexity_over_time.png}
    \caption{Communication Perplexity tracked over a training run of AI Mother Tongue when using SQ-VAE trained across three runs, and a 95\% confidence interval is shown.}
    \label{fig:comm-perplexity-over-training}
\end{figure}

At the start of training, the agents are exploring all possible outputs, leading to a uniform random sampling across the majority of the codebook.
As the RL begins to solve the asymmetric information task (between step 3,000 and 10,000), communication perplexity collapses and stabilises at $\sim 3.0$.
This collapse represents spontaneous semantic compression.
Instead of using the full theoretical capacity of the codebook, a small subset of codes is assigned useful semantic meaning.

\subsection{Codebook Utilisation}
\label{sec:4.5.2}

To further understand the agent's use of language, the frequency of each code is analysed.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{vocab-frequency.png}
    \caption{Normalised frequency of each code across 2880 runs}
    \label{fig:vocab-frequency}
\end{figure}

From Figure \ref{fig:vocab-frequency}, the distribution exhibits a strong skew, giving a low entropy score of 2.98 and a high Gini coefficient of 0.78.
When visualising this empirical distribution with a rank-frequency plot, it demonstrates exponential decay closely resembling Zipf's Law, especially for the first 25 codes.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{rank-frequency-vs-zipf.png}
    \caption{Vocabulary rank-frequency plot compared to Zipf's Law}
    \label{fig:rank-frequency-zipf}
\end{figure}

However, a notable deviation from Zipf's Law does occur at the tail-end of the distribution where the frequency rapidly decreases.
Although a sudden drop to zero is not a feature of a classic Zipfian distribution, it is a feature of the truncated Zipfian distribution \cite{TruncatedZipfianDist}.
This occurs because in most languages analysed, an infinite number of words can be created; however, in this system, there only exists a finite set of codes.
The emergence of this phenomenon suggests the agents were successful at assigning high-frequency tokens to common semantic meaning and low-frequency tokens to unnecessary tokens.

\section{Cognitive Mapping and Spatial Memory}
\label{sec:4.6}

\subsection{Memory Probing}
\label{sec:4.6.1}

To understand what the agent concretely knows, six strict linear probes were developed following Guillaume et al.'s methodology \cite{Probing}.
Memory is tested for allocentric memory (information is stored using a global coordinate system) and egocentric memory (information is stored relative to the self).
Each probe takes as input the final layer of the Long-Short Term Memory (LSTM) module, and predicts one of the following using MSE loss:
\begin{itemize}
    \item The agent's own next target's coordinates (tests for allocentric memory)
    \item The agent's partner's next target's coordinates(tests for allocentric memory)
    \item The agent's own next target's relative coordinates from the agent's coordinates (tests for egocentric positional memory)
    \item The agent's partner's next target's relative coordinates from the agent's coordinates (tests for egocentric positional memory)
    \item A unit vector in the direction of the agent's next target (tests for egocentric directional memory)
    \item A unit vector in the direction of the agent's partner's next target (tests for egocentric directional memory)
\end{itemize}

Initially, in a simulation run, agents have no way of knowing the location of their targets.
To account for this, the models are trained only on the last half of the timesteps to ensure the LSTM contains the necessary information for training.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{lstm_probing.png}
    \caption{LSTM Probing Models Test Loss over simulation runs with various tasks}
    \label{fig:lstm-probing}
\end{figure}

The relative distance and unit direction models are shown to never improve over the course of a simulation, suggesting the LSTM does not follow egocentric memory.
However, the absolute location model shows that the LSTM uses allocentric memory.

Even though the absolute location model does show the LSTM storing some information, the model converges on an MSE of 1.6.
This error represents an uncertainty radius of 1.6 cells, showing how the memory acts as a 'regional beacon' for the agent's targets.

An important conclusion from figure \ref{fig:lstm-probing} is that the agent's need for communication decreases as the simulation runs, and they store target locations in the LSTM module.

\subsection{Action and Communication Entanglement: $P(\text{Action}_{self}^{t} | \text{Code}_{self}^t)$}
\label{sec:4.7.3}

To understand the relationship between the agent's code and its own action, a confusion matrix is made:

\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{own-action-given-code.png}
    \caption{Probability of agent's action given emitted code within first four simulation steps}

The figure \ref{fig:own-action-given-code}, which calculates the conditional probability of the next action given a code, is unlike \ref{fig:partner-action-given-code}.
Specific codes lead to strong probabilities of certain movement actions.
This is a symptom of state-action entanglement.
The action head and communication head for the actor are simple, and as they are both derived from the LSTM module, they are strongly correlated.
This leads to the code being output by the actor, inadvertently encoding its next action choice.

\subsection{Context-Aware Signalling (Inventory-Dependent Language)}
\label{sec:4.7.4}

To understand if the agents are using a pragmatic language, the conditional probability of token usage given the ingredients it has is calculated.

\begin{figure}[H]

\section{Evaluation of Success Criteria}

\paragraph{C1 - High-Throughput Environment}
C1 required a configurable, vectorised Rust environment with Python bindings, achieving at least 50K environment steps per second and supporting real-time visualisations.
Section \ref{sec:3.4} detailed how this was implemented.
The final system was able to reach over 200K environment steps per second on my hardware, as shown in section \ref{sec:4.2.1}.
C1 was fully achieved.

\paragraph{C2 - MARL Framework}
C2 required a functioning multi-agent reinforcement learning implementation with a shared actor and a centralised critic capable of learning the complex environment.
Section \ref{sec:3.3} detailed how this was implemented using MAPPO.
The final system with no communication was able to reach $\sim 40\%$ of the mean task reward of the expert agent, as shown in section \ref{sec:4.4.1}.
C2 was fully achieved.

\paragraph{C3 - Interchangeable Communication Protocols}
C3 required at least two protocols (no communication and discrete communication) and the implementation of the AI Mother Tongue architecture suitable for this setup, which are all interchangeable within the same training pipeline.
The communication strategy pattern (as detailed in section \ref{sec:3.3.5}) demonstrated that protocols can be interchanged easily.
The MARL baseline comparison in section \ref{sec:4.4.1} could not be implemented without the hot-swapping of protocols.
C3 was fully achieved.

C4 was fully achieved.

C5 required the framework to be reusable for other researchers.
