\subsection{Simulation}
\label{sec:2.1.1}

To evaluate communication, we investigate through the following cooperative game.

There is an $M x N$ board.
The game lasts for $T$ timesteps and at every timestep, every agent simultaneously does one action.
The goal of the game is for the agents to cooperate and complete a required recipe.

\subsection{Stations}
\label{sec:2.1.1.1}
Stations are dotted randomly across the map and are immovable throughout the course of the game.
Each station has an assigned ingredient. The station can either be of type 'pick up' or 'drop off'.
A 'pick up' station only allows the agents to take ingredients from them.
A 'drop off' station only allows the agents to give the station the assigned ingredient.

\subsection{Agents}
\label{sec:2.1.1.2}

Each agent is two sets of ingredients - an input set and an output set.
Their input set determines which ingredients the agents need to collect.
Once they have all of their ingredients, they get removed from their inventory and replaced with their output ingredients.
The agents can collect ingredients from 'pick up' stations or other agents.
The agents can drop off their ingredients at 'drop off' stations.

One important feature of this game is the agents have limited range of sight and can only view the nearest entity at any given timestep.
This converts this game from a Markov Desicion Process to a Partially Observable Markov Decision Process.
The agents must work together to discover the required stations and use them to complete recipes.

At every timestep, every agents takes a single action.
Agents have five possible actions: move north, move south, move west, move east and interact.
The first four of which move the agent one step in the given direction.
If agent A executes the 'Interact' action:
\begin{enumerate}
    \item If visible agent B has a required item, place in A's inventory, and remove from B's inventory.
    \item If visible 'pick up' station has a required item, place in A's inventory.
    \item If visible 'drop off' station wants an item we have, remove from A's inventory.
\end{enumerate}

\subsection{Success Criteria}
\label{sec:2.5.1}

\begin{enumerate}
\item Design and implement a simulation to train and demo multi-agent interactions. The simulation should be able to have single runs visualised. The simulation should control the state of every agent and collect data.
\item Implement a functional MAPPO-based multi-agent reinforcement learning (MARL) system capable of learning, given a simulation program with set methods.
\item Develop a communication protocol that allows agents to transmit raw data between each other.
\item Similarly, develop another communication protocol that uses a VQ-VAE to transmit vectors between agents.
\item The training and evaluation pipeline should be able to train and evaluate models automatically and allow interchangeably using both communication protocols.
\end{enumerate}

\subsection{Extension}
\label{sec:2.5.2}

\begin{itemize}
\item Implement a third communication protocol using a hierarchically quantised variational autoencoder (HQ-VAE). This method was suggested by the AI Mother Tongue paper as a possible future work.
\item Add hyperparameter tuning for my discrete communication protocol. This can involve the language size, quantisation levels, and message frequency to optimise agent coordination and learning efficiency.
\item Add a new task where agents must solve for objectives other than an assembly line (e.g., coordinated navigation, resource gathering, or adversarial avoidance), enabling analysis of communication protocol effectiveness in more diverse MARL settings.
\end{itemize}

\subsection{Tool Selection \& Justification}
\label{sec:2.6.2}

% After reviewing similar projects, I settled on separating my environment and my from my controller.
Python was chosen as the programming language for the controller, primarily due to its excellent library support for machine learning and data analysis
(including but not limited to NumPy ¬R, Pandas ¬R, Matplotlib ¬R and Jupyter Notebooks ¬R).
PyTorch ¬R was chosen to develop the models due to its high performance and imperative execution model, which facilitates rapid iterative prototyping.
However, Python is not a performance language and, as of development ¬R, has a GIL ¬R meaning executing simulations of my environment must be effectively done serially, bottlenecking my performance heavily.
I chose Rust to develop my environment for its high performance and memory safety allowing for rapid development.

To streamline my package management and environment integration to the controller, I used a pyproject.toml file.
This file handled installing packages from external sources and also building my rust environment program to allow my Python controller to access the environment.

For version control, I used GitHub to store my git repository which included both my program and my dissertation.
I also regularly updated the version of code on my laptop as an extra backup.
